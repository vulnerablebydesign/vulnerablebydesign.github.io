The Mission Briefing

There was nothing to break into. No VPN, no port scan, no shell to catch. This was an HTB Academy module — JavaScript Deobfuscation, module 14 of 15 on my way to the Defense Operations Analyst certificate. Each section spins up its own little sandbox, and the only adversary is a wall of code that someone deliberately scrambled so a human couldn't read it.

Which, when you think about it, is exactly what malware does. An attacker who drops a malicious script onto a page doesn't leave it readable for the SOC analyst who shows up later. They pack it, mangle it, base64 it, and bury the payload under generated garbage. Learning to unwrap that is the whole game.

The lesson the whole module circles back to is simple enough to fit on a sticky note: everything client-side belongs to you. HTML, CSS, and JavaScript are all delivered to your browser in cleartext. If the server sent it to render the page, you can read it. The only question is how much work the author made you do first.

Code Obfuscation: Why Anyone Would Do This

Before deobfuscating, the module explains the why. Obfuscation makes code hard for a human to read while keeping it functionally identical (usually a bit slower). The tools build a dictionary of tokens and rebuild the program at runtime. JavaScript is the prime target precisely because it ships to the client in cleartext — there's nowhere to hide.

The use cases: protecting intellectual property, (badly) hiding client-side "auth" or crypto, and — the one that matters for a SOC analyst — malware dodging IDS/IPS signatures. If the malicious string never appears in plaintext, naive signature matching never fires.

GlaDOS
"They obfuscate to hide intent from the very systems built to catch them. It works, too — until someone with patience and a beautifier comes along. That someone is, regrettably, you."

Deobfuscation: Unpacking the Blob

This is where the actual technique lives, and the module breaks it into three escalating tiers.

1. Beautify the minified one-liners. Minified code is just whitespace-stripped — functionally readable, visually hostile. Firefox's debugger has a Pretty Print button ({ }), or you can paste into Prettier / Beautifier. Per the exercise's recorded tooling, the sites in play were prettier.io and beautifier.io.

2. Deobfuscate packed code. The classic eval(function(p,a,c,k,e,d){...}) wrapper is the Dean Edwards packer. You run it through an UnPacker (the exercise used matthewfl.com/unPacker.html), or — my favorite trick — you replace the final eval with console.log so the code prints its payload instead of executing it.

3. Manual reverse engineering for heavy or custom obfuscation, where no tool will save you.

The task was to deobfuscate a secret.js and recover its flag. The trap here: the module's section text shows an illustrative packed example with the real value redacted as ...SNIP.... Copy that and you get nothing. You have to actually run the deobfuscation on the exercise's own file.

Code Analysis: Reading the Function

With the script beautified, I read the deobfuscated generateSerial() line by line. The interesting part: it builds an XMLHttpRequest, opens a POST to /serial.php, and calls send(null) — an empty body. No data, no return handling. It's a function that exists but appears never to be wired up to anything in the UI.

That's the tell. An unreleased or unused function is exactly the kind of forgotten doorway worth knocking on. The methodology the module pushes is good habit: when you hit an unfamiliar API, Google it, understand what it does, then infer the author's intent. Replicating that empty POST server-side might expose functionality the developers never meant to ship.

GlaDOS
"A function that talks to an endpoint nobody calls. How tantalizing. The developer left a door, forgot to lock it, and then forgot the door existed. People are so reliably forgetful. It's the only thing I can count on."

HTTP Requests: Replicating the POST

The function made an empty POST, so I made an empty POST. The module teaches just enough curl to do it: a bare curl URL is a GET (returns HTML), -X POST changes the method, -d "k=v" attaches body data, and -s silences the progress noise for clean output.

curl -s http://TARGET:PORT/serial.php -X POST ****************************N2gxNV8xNV9hX3MzY3IzN19tMzU1NGcz

The server answered the empty POST with an encoded blob — not a flag, but clearly something. (The "do the exercise, don't copy and paste ;)" note in the section text is a decoy; the real response comes from your own target.) That string has all the fingerprints of base64: alphanumeric with the base64 alphabet, length a clean multiple of four. Time to decode.

Decoding: From Blob to Secret

The decoding section is a tour of the three encodings you'll meet constantly inside obfuscated JS, and how to spot each on sight.

Fact Sphere
"Base64 was standardized in RFC 4648. It maps three bytes to four ASCII characters using a 64-symbol alphabet, which is why valid base64 length is always a multiple of four, padded with '=' as needed. Hexadecimal uses only 0–9 and a–f. ROT13 is a Caesar cipher with a fixed shift of thirteen — applied twice, it returns the original. None of these are encryption. Encoding is reversible without a key. Confusing the two is a common and fatal mistake."

The cheat sheet the module hands you:

  • base64 — alphanumeric plus +/=, length a multiple of 4. Decode with base64 -d.
  • hex — only 0-9a-f. Decode with xxd -p -r.
  • rot13 / Caesar — shifted letters. Decode with tr 'A-Za-z' 'N-ZA-Mn-za-m'.

I decoded the blob from the previous step:

echo 'N2gxNV8xNV9hX3MzY3IzN19tMzU1NGcz' | base64 -d 7h15_15_a_s3cr37_m3554g3

"This is a secret message." Now I had the decoded value the server wanted. The final move was to POST it back — the same /serial.php endpoint, this time with the decoded string as a serial= parameter.

curl -s http://TARGET:PORT/serial.php -X POST -d "serial=7h15_15_a_s3cr37_m3554g3" The flag is a lieHTB{ju57_4n07h3r_r4nd0m_53r14l}

The Full Chain: Skills Assessment

The Skills Assessment ties every section into one kill chain. The scenario: a pentest of a web server using JavaScript and APIs, and I had to determine its functionality end to end. Same loop, harder target.

I started by enumerating the JavaScript in use — a minified api.min.js — and running it to see its response, which surfaced the first flag of the chain. Then I deobfuscated the file, found a flag variable inside, beautified the logic, and read it the way I'd read generateSerial() earlier.

The deobfuscated code pointed at a request I needed to replicate to retrieve a secret key. This time the server's response was hex, not base64:

echo '4150495f70336e5f37333537316e365f31355f66756e' | xxd -p -r API_p3n_73571n6_15_fun

What I Learned

  • Client-side code is always readable — HTML, CSS, and JS are delivered to your browser in cleartext. View Source and the debugger are recon tools, not just dev tools. External <script src> files are fetchable too.
  • Obfuscation is reversible, not secure — tools build a token dictionary and rebuild at runtime. Beautifiers handle minification; UnPackers (or swapping eval for console.log) handle the Dean Edwards packer.
  • Read functions for intent, not just syntax — an unused generateSerial() that POSTs to /serial.php with an empty body is a forgotten doorway worth replicating server-side.
  • curl is the deobfuscator's hands — -X POST for method, -d for body, -s for clean output. If the JS makes a request, you can make it yourself.
  • Spot the encoding on sight — base64 (alphanumeric, length ÷ 4), hex (0-9a-f), rot13 (shifted letters). Encoding is reversible without a key; encryption is not. Decode with base64 -d, xxd -p -r, or tr.
  • Don't copy the course example — the redacted ...SNIP... illustrations and decoy responses are traps. Run the technique on your own target's file every time.
  • This is the malware-triage loop — locate JS, recognize obfuscation, beautify, analyze, replicate, decode. The same workflow a SOC analyst runs against a real malicious script.
