The Empty Bowl

The brief: "The R4 back-pantry (Snack Hill Conquest, Sep–Nov 2025) mixes a fresh loaf (cfg = {}) each attempt. Every loaf inherits from the mother starter. The oven checks two fields before it pulls the tray."

The handler creates a fresh empty object and reads two fields off it:

document.getElementById('proto-unlock-btn').addEventListener('click', async () => { const cfg = {}; if (cfg.authorized !== true) { setStatus('proto', 'err', 'cfg.authorized is not true — pollute the prototype'); return; } if (!cfg.r4VaultPhrase) { setStatus('proto', 'err', 'cfg.authorized ok, but cfg.r4VaultPhrase missing'); return; } await handleUnlock('proto', 'sourdough_starter', cfg.r4VaultPhrase); });

An empty object literal inherits from Object.prototype. So cfg.authorized is undefined — unless something has assigned to Object.prototype.authorized. Same for cfg.r4VaultPhrase. The hint card on the page literally instructs: "pollute Object.prototype to get values into cfg".

Fact Sphere
"Prototype pollution was first publicly catalogued in 2018. JavaScript has inherited from Object.prototype across approximately 99.7% of operational deployments since 1995. The correlation between framework convenience and prototype-mutation vulnerabilities is considerable. The most-exploited gadget remains lodash.merge on an attacker-controlled JSON payload."

The Problem That Tripped Most Players

Polluting Object.prototype.authorized = true is trivial. The harder question: what value goes into r4VaultPhrase? The handler sends cfg.r4VaultPhrase as the keyCandidate. The server hashes whatever string it receives. So the player must send the specific phrase the server is hashing against.

The phrase is not displayed in the UI. The brief says so explicitly. The page's inspector for stage 5 shows r4VaultPhrase: undefined. The challenge brief is the entire roadmap — "R4 back-pantry, Snack Hill Conquest, Sep–Nov 2025" — but the phrase needs a derivation rule.

The Rule Was Hidden In Plain Text

The stage-5 inspector card on /admin/ has a row I almost missed on the first pass:

Seal format: R<round>-<EVENT-SLUG>-VAULT-<YEAR> This oven sealed at: R4 closing checkpoint · Snack Hill 2025

That's the entire derivation rule. R+round, dash, the event slug in UPPER-CASE-WITH-HYPHENS, -VAULT-, year. Plug R4 + "Snack Hill" + 2025:

R4-SNACK-HILL-VAULT-2025

The hyphen-between-words style is anchored by stage 02's pre-known key (R2-MUNCHIES-2025-HRC-LOUISVILLE — words hyphenated, even where ambiguous). First guess on the seal:

What I Learned

"Hidden phrase" challenges have a hint somewhere on the page. The brief said the phrase wasn't visible in the UI. That was true of the phrase itself. The format of the phrase, plus the round metadata that fills the format, was sitting in an inspector card three lines above the unlock button. The CTF author was generous; the bug class was prototype pollution but the gating mechanism was schema-following.

If the server hashes the keyCandidate, the JS-side ceremony is optional. Stage 5's intended path forces the player to construct an attack JSON, set Object.prototype.authorized=true, set Object.prototype.r4VaultPhrase=..., and click. The server doesn't see any of that — it sees one HTTP body containing one string. Skipping the ceremony is honest about the actual trust boundary.

For the writeup, walk the intended path anyway. A complete writeup explains why the bug class exists. Skipping pollution and going straight to the POST gets the flag but leaves the reader without the lesson. The intended attack — Object.prototype.authorized=true; Object.prototype.r4VaultPhrase='R4-SNACK-HILL-VAULT-2025'; document.getElementById('proto-unlock-btn').click() — is a one-line DevTools-Console pollution that demonstrates the principle. Worth doing once, even after the flag is in hand.
