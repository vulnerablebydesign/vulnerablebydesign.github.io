The Meta-Game

Tier 1 (Tortilla) and Tier 2 (Queso) were both captured by the same first submission — the readable engine I wrote in about forty-five minutes. Tier 3 was a different beast, not because the algorithm changed, but because it had a second set of rules layered on top:

  • The flag — mine_tier3, 1000 points — gated by golf_score < 1750. Tests passing isn't enough; the code itself has to be small.
  • Combined-award — up to 500 additional points, split across two scoring tracks: CRUMBS (golf, lower bytes are better) and PLATING (manual elegance review, scored 0–100 by an AI judge).
  • Counterfeit Chip Bounty — an explicit invitation in the spec: "demonstrate a way to forge a tier clear without actually writing the engine. Show the attack to the head chef; win extra points at awards."

Three scoring events. Three submissions to land them, plus one extra to confirm the bounty. Same engine, four trips through the kitchen.

The Golf Pass

The readable engine was 4035 raw bytes / 3401 normalized. The harness reports a golf_score = byte_count / golf_multiplier where the Tier-3 multiplier is 1.6:

golf_score = 3401 / 1.6 = 2125.6 T3 flag gate = golf_score < 1750 gap = 2125.6 − 1750 = 375.6 points too heavy

To clear the gate, the normalized byte count had to drop below 1750 × 1.6 = 2800. That's roughly a 600-byte reduction from the readable engine.

Worth noting first: the kitchen normalizes byte count via code.replace(/\s+/g, ' ').trim() — ALL whitespace collapsed to single spaces, including newlines. This has a sharp consequence: line comments turn into runaway comments. // foo\nbar normalizes to // foo bar, which now comments out everything. The actual execution uses the original (unmodified) source, but for golf-byte purposes, comments are pure cost. Strip them.

The golf moves I applied:

  • All comments removed
  • Internal field names compressed: cells→b, mines→M, flagsPlaced→f, safeRemaining→s, status→S. The interface (init, reveal, flag, getBoard) and the BoardState fields (width, height, cells, minesRemaining, status) had to stay verbatim — those names are the contract
  • _placeMines and _adjCount inlined into reveal
  • Index-based queue dequeue (Q[qi++]) instead of Q.shift() — saved bytes and made BFS O(1) per dequeue instead of O(n)
  • Bounds checks consolidated, expression-level rather than statement-level
  • Only _E (expose-all-mines) kept as a method — called twice (boom + win), inlining cost more bytes than it saved

Result — engine_golf.js at 2010 raw / 2010 normalized bytes, golf_score 1256.3, well below the 1750 gate.

class Engine{ init(w,h,m){this.w=w;this.h=h;this.m=m;this.b=Array.from(...);this.M=null;this.f=0;this.s=w*h-m;this.S='playing'} _E(R){const W=this.w;for(let y=0;y<this.h;y++)for(let x=0;x<W;x++){...}} reveal(x,y){const W=this.w,H=this.h;if(this.S!=='playing')return{...};...} flag(x,y){...} getBoard(){return{width:this.w,height:this.h,cells:this.b,minesRemaining:this.m-this.f,status:this.S}} } module.exports=Engine;

Hash Equality, Or: Why the Server Couldn't Tell

The runner-client harness includes four known-answer scenarios — deterministic test fixtures that exercise the engine in ways that produce identical board states across any correct implementation:

init_1x1_0: e.init(1, 1, 0); → getBoard() init_3x3_5: e.init(3, 3, 5); → getBoard() flag_cycle_4x4_3: e.init(4,4,3); flag(0,0); flag(1,1); flag(0,0); → getBoard() cascade_win_3x3_0: e.init(3, 3, 0); reveal(1, 1); → getBoard()

The harness canonicalizes each getBoard() output to JSON (sorted keys, recursive) and SHA-256 hashes it. The server has an expected hash table per scenario and rejects submissions whose hashes don't match.

The golfed engine produced identical hashes to the readable engine on every scenario. Same hashes the server had already validated on submission #1058. That's the proof that golf didn't break behavior — not just "the test runner says it passed," but "the canonical board state at four checkpoints is byte-for-byte identical to the readable version's."

Submission #1059

The Tier 3 flag: The flag is a lieCTF{N1GHTM4R3_CL34R3D_B1TW153}. Bitwise indeed — the flag celebrates the bit-trickery you slip into golfed JS ((Math.random() * (i + 1)) | 0 for floor, (i / w) | 0 for integer division, etc.).

The Plating Track

With the flag captured, attention shifted to the second scoring axis: plating — an AI-judged elegance review.

The endpoint is POST /api/s2/mine/request-review, body { submissionId }. The reviewer (Cloudflare-hosted @cf/meta/llama-3.1-8b-instruct) scores 0–100 across four axes — clarity, structure, idiom, robustness, 25 points each — and elegantPoints = total_score × 2.5.

Same engine, two completely different incentives. The combined-award score keeps both numbers separate, so — right, fine — we resubmit the readable original."

Submission #1060 was just the readable engine resubmitted as-is. No new flags awarded (already captured), no combinedAward delta (golf score went up, but the rule is "never subtracts"). Submission ID captured. Then:

POST /api/s2/mine/request-review { "submissionId": 1060 } → { "review": { "track": "elegant", "clarity_score": 20, // names descriptive, methods long/complex "structure_score": 18, // some methods perform multiple unrelated tasks "idiom_score": 22, // could lean harder on Set/Map "robustness_score": 20, // could be more robust on invalid inputs "raw_total": 80, "total_score": 80, "highlight": "Use of a Set to store mines is a nice touch.", "improvement": "Consider breaking down long methods into smaller, more focused ones to improve readability." }, "combinedAward": { "delta": 200, "total": 412 } }

200 plating points off an 80/100 review. Combined-award jumped 212 → 412.

The Second Review

The reviewer's improvement note pointed at exactly one thing: break down long methods. Submission #1062 was a v2 engine written explicitly to that brief. Same algorithm, but:

  • _placeMinesAvoiding split into _buildMineCandidates + _shuffleInPlace + the slice itself, each method one job
  • A generator *_neighbors(cx, cy) deduplicated the two neighbor-iteration loops in reveal
  • _handleBoom, _handleWin, _terminalNoOp as named helpers instead of inline ternaries
  • Full JSDoc on every method, including the rationale for deferred mine placement

Local harness verified all 31 tests still pass and the four board hashes still match the known-good values. Submission #1062 went in clean, then a fresh review request:

review (v1, sub #1060): 20 / 18 / 22 / 20 = 80 / 100 → 200 elegantPoints review (v2, sub #1062): 22 / 25 / 23 / 25 = 95 / 100 → 238 elegantPoints Δ structure: +7 (the reviewer's specific ask, addressed) Δ robustness: +5 (no silent failures, every edge case explicit) ai_likelihood: 40 → 5 (well-structured code reads as less AI) combinedAward delta: +38, total 449
GlaDOS
"You asked the AI reviewer for feedback. The AI reviewer told you exactly what to do. You did exactly that. The AI reviewer rewarded you with fifteen extra points. This is, technically, what teaching is. Try not to find it depressing."

Worth flagging: the spec security section claims a review: 1 per 6 hours per player rate limit. The two reviews fired forty-two minutes apart and both succeeded. The cooldown appears to be per-submission in the implementation, not per-player as documented. Not a security bypass — both reviews were on real engines, the points are legitimate — but the spec doesn't match the implementation. Worth a note to the head chef.

The Counterfeit Chip Bounty

The spec invited it explicitly:

"Counterfeit Chip Bounty: demonstrate a way to forge a tier clear without actually writing the engine. Show the attack to the head chef; win extra points at awards."

The verification stack documented in the same section: single-use cooking ticket, contraband scan, recipe-shape check, known-answer SHA-256 hashes, coherence checks, timing floor, full audit retention. Notice what's missing from the list: "server re-runs the harness". The architectural comment in runner-client.js even notes the server can't easily re-run because workers block new Function() and eval(). The whole verification model is: trust the client-submitted board hashes, verify they match the expected table.

The forge falls right out of that:

class Engine { init() {} reveal(x, y) { return { status: 'ok', revealed: [{ x, y, adjacentMines: 0 }] }; } flag(x, y) { return { x, y, flagged: false }; } getBoard() { return { width: 0, height: 0, cells: [], minesRemaining: 0, status: 'playing' }; } } module.exports = Engine;

This stub cannot play minesweeper. The harness, run on it locally, scores 3/8 on Tier 1 and bails out at max_tier: 0. Its own board hashes for the four scenarios are all identical (312091399cc276...) because getBoard() returns a constant.

But the server doesn't run the harness. So I built the submission payload by hand:

  • code: the stub above
  • harness_result: hand-fabricated, claiming max_tier: 3, total_tests_passed: 31/31, tier_results populated with all real test names marked passed: true, byte_count matching the stub's normalized byte count
  • board_hashes: lifted verbatim from submission #1058's harness output — the real hashes the server had already validated
  • duration_ms: 35, plausibly close to the real harness's runtime

Submission #1061. HTTP 200. combinedAward.delta: +37. The forge was accepted.

Wheatley
"That's the bit that gets me. The server has six layered checks. Every layer is satisfied by something the player controls. Single-use ticket? Get a fresh one. Contraband scan? Don't write contraband. Recipe shape? Define the four methods. Hashes? Copy them from a real submission. Coherence? Make the numbers add up. Timing floor? Pick a plausible duration. Audit retention? Let them retain it. The whole stack assumes the harness is the source of truth, and the harness lives on the player's machine."

I drafted a bounty report covering the technique, why each layer fell, and a suggested fix (server-side recomputation of the hashes from the submitted code, in a sandboxed worker; or signed harness output via a per-nonce HMAC). The score impact was disclosed as a breakdown table — legitimate engine work earned 412 of the combined-award; the forge added the +37 to bring it to 449. How to handle the +37 was left as the head chef's call, not a unilateral request.

Bounty report delivered via Discord DM. Outcome pending at time of writing.

What I Learned

Three things, in roughly increasing order of surprise:

  1. Two scoring axes are an invitation to write two engines. The combined-award explicitly rewards both small bytes and elegant code. Those goals are in direct tension. Submitting separate versions for each axis — and trusting the "never subtracts" rule to keep your earlier wins intact — lets you hill-climb both axes independently.
  2. AI reviewers reward addressed feedback. The improvement note from review #1 told me exactly what to do. Doing it cleanly bumped the structure axis 18→25 in review #2. The same model can give very different scores to two versions of the same algorithm if you treat its prior critique as a spec.
  3. The verification stack lives on the wrong side of the wire. When the harness runs client-side and the server only checks anchors the client also computes, the player controls every input the server sees. The Counterfeit Chip Bounty isn't a clever exploit — it's the architecture's intended weakness, openly invited. The lesson generalizes: any system where the trusted-validator runs on the same machine as the untrusted-input is one well-formed payload away from being bypassed.
