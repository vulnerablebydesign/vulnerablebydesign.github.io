Redacted In The Wrong Place

The brief: "Closing-shift log from R1→R4 (2025) with one early-S2 entry. A policy redactor hides anything that looks like a keyphrase when rendering — but the raw entries were already loaded into the page."

The brief tells you exactly where the bug is. The redactor runs at render time, not at load time. The raw data made the trip from server to browser intact; only the visual presentation strips it.

GlaDOS

What I Learned

Client-side redaction is presentation, not security. If the value made the network trip from server to browser, it has been disclosed. Any client-side filter applied after that point is a UX feature, not a confidentiality control. Real redaction happens at the source — the server stripping the value before it leaves the API boundary.

Globally-exposed arrays are a gift. window.__webHexLogs = LOG_EVENTS is the kind of thing developers write to make debugging easier; it's also the kind of thing that turns a "find the redacted entry" puzzle into a one-liner. Watch for window.__ prefixed exports, data-* attributes, hidden form fields, and inlined JSON blobs in <script type="application/json"> tags — same pattern, every time: the data was already shipped.
