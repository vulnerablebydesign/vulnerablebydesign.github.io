The Stamp That Didn't Cover Everything

The brief: "A loyalty card was issued to a viewer, but the signature stamp only covers handle|exp. The tier-field upgrade was deferred to a sprint that never happened."

That last clause is a confession. The developer wrote a JWT-shaped token, MAC'd it covering only two fields, then ran out of time before extending the MAC to the third (the one that actually mattered for authorization). This is a real bug class — incomplete signature scope — and it's the most common "I rolled my own JWT" failure mode short of accepting alg: none.

Reading The Decoy

The challenge baked the decoy token directly into the page. Pulled from /admin/ source:

Three dot-separated b64url segments. Decoded:

header = {"alg":"HS256","typ":"SNK"} // typ=SNK is flavor; behaves like JWT payload = {"handle":"FirstOrganizer","exp":9999999999,"role":"viewer"} mac = HMAC-SHA256(secret, "handle|exp") // does NOT cover role

The on-page inspector even labels it: "Signature covers: handle | exp (role is NOT covered)". The signing secret S2_SIGNING_2025 is also openly displayed — but it's a red herring for this attack. We don't need to resign; we need to keep the existing MAC valid while changing a field outside its scope.

The Forgery

Flip role from viewer to admin, leave handle and exp untouched, re-encode the payload, splice the same MAC back. Single Python block:

What I Learned

MAC scope is the whole game. When you see "the signature only covers X" in a challenge brief, the bug class is fixed: X is the integrity-protected slice; everything else in the same token is forgeable. JWT-shaped tokens make this especially easy to fall into — the spec encourages thinking of "the payload is signed" when in reality only the bytes the implementer chose to feed into the HMAC are signed.

The signing secret was a red herring. S2_SIGNING_2025 sat right there on the page begging to be used. But knowing the secret only matters if you need to produce a new MAC. When the existing MAC's scope already covers the bytes you're not changing, you don't need to compute anything — you just preserve the ciphertext. Every published "JWT secret" challenge gets harder if you assume the secret is irrelevant; nine times out of ten you don't need it.

Read the JS bundle for the verifier shape. The handler's last line — handleUnlock('token', 'loyalty_card', `admin-${payload.handle}-${payload.exp}`) — revealed that the server only ever sees the derived string, not the token. Skipping the JWT and POSTing the derived string was strictly cheaper. Reading the verifier always pays.

  1. Pull decoy token from /admin/ HTML — handle=FirstOrganizer, exp=9999999999, role=viewer
  2. Read JS handler — MAC covers handle|exp only; server-side keyCandidate is admin-{handle}-{exp}
  3. Verify MAC scope locally (HMAC-SHA256 with S2_SIGNING_2025 matches the token's MAC)

  5. Skip the token roundtrip: POST keyCandidate directly → flag in response
