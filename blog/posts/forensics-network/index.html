Two Captures

The challenge dropped two files into my lap: a 60KB pcap labeled station-traffic.pcap and a 15KB zip file that immediately rejected every password I threw at it. The zip header confirmed AES-256 encryption — no brute force, no known-plaintext shortcuts. Whatever was inside that archive, the key was hiding somewhere in Stage 1.

Two files. Two stages. The pcap was the door and the zip was the room behind it.

Credential Harvest

I opened the first pcap in tshark and filtered for HTTP traffic. The capture was clean — unencrypted HTTP requests between a handful of internal station hosts. Three POST requests stood out immediately, each hitting a different authentication endpoint:

$ tshark -r station-traffic.pcap -Y "http.request.method == POST" -T fields \ -e http.host -e http.request.uri -e http.file_data 10.42.0.5 /sso/forum/login dXNlcm5hbWU9bS5jcnVtYiZwYXNzd29yZD1odW50ZXIyLXNuYWNrbmV0 10.42.0.12 /admin/auth dXNlcm5hbWU9b3BlcmF0b3ImcGFzc3dvcmQ9c25ja3c0djNfYWRtMW5fMjA0Mg== 10.42.0.8 /roster/v1/auth dXNlcm5hbWU9ai5xdWFydGVybWFzdGVyJnBhc3N3b3JkPVJPU1RFUi1URU1QLTIwNDItMDktMDM=

Base64. Not encryption — encoding. Three credentials transmitted in plaintext over HTTP, waiting to be read by anyone with a packet sniffer:

$ echo "dXNlcm5hbWU9bS5jcnVtYiZwYXNzd29yZD1odW50ZXIyLXNuYWNrbmV0" | base64 -d username=m.crumb&password=hunter2-snacknet $ echo "dXNlcm5hbWU9b3BlcmF0b3ImcGFzc3dvcmQ9c25ja3c0djNfYWRtMW5fMjA0Mg==" | base64 -d username=operator&password=snckw4v3_adm1n_2042 $ echo "dXNlcm5hbWU9ai5xdWFydGVybWFzdGVyJnBhc3N3b3JkPVJPU1RFUi1URU1QLTIwNDItMDktMDM=" | base64 -d username=j.quartermaster&password=ROSTER-TEMP-2042-09-03

Three users. Three passwords. A forum SSO, an admin panel, and a roster system. All of them naked on the wire.

Opening the Archive

Three passwords, one locked zip. The forum credentials didn't work. The roster password didn't work. But the admin password — snckw4v3_adm1n_2042 — cracked the AES-256 archive open cleanly:

$ 7z x encrypted-stage2.zip -psnckw4v3_adm1n_2042 Extracting archive: encrypted-stage2.zip Everything is Ok Size: 84291 Compressed: 15402 $ file stage2-exfil.pcap stage2-exfil.pcap: pcap capture file, microsecond ts (little-endian) - version 2.4

Another pcap. Stage 2 was another packet capture, and whatever was inside it had been worth encrypting.

The DNS Channel

The second pcap was small — 548 packets — but its protocol hierarchy told the whole story at a glance:

$ tshark -r stage2-exfil.pcap -z io,phs -q =================================================================== Protocol Hierarchy Statistics Filter: eth frames:548 ip frames:548 udp frames:548 dns frames:548 ===================================================================

Every single packet was DNS. No HTTP, no TCP, no ICMP. Pure DNS over UDP. This was DNS exfiltration — data smuggled out through domain name queries, one of the oldest tricks in the network forensics playbook.

I filtered for the query names:

$ tshark -r stage2-exfil.pcap -Y "dns.qry.name contains orbital-pantry" \ -T fields -e dns.qry.name | sort -u s00.orbital-pantry.net s01.orbital-pantry.net s02.orbital-pantry.net s03.orbital-pantry.net s04.orbital-pantry.net s05.orbital-pantry.net s06.orbital-pantry.net s07.orbital-pantry.net s08.orbital-pantry.net s09.orbital-pantry.net s10.orbital-pantry.net s11.orbital-pantry.net

Twelve queries to orbital-pantry.net, sequenced s00 through s11. Each subdomain label carried a 4-character payload. The sequence numbers made the reassembly order obvious.

Decoding the Signal

I extracted just the subdomain labels, stripping the domain and the sequence prefix to isolate the payload characters:

$ tshark -r stage2-exfil.pcap -Y "dns.flags.response == 0 && dns.qry.name contains orbital-pantry" \ -T fields -e dns.qry.name | sort -t. -k1 -V | \ sed 's/\.orbital-pantry\.net//' | sed 's/^s[0-9]*//' | tr -d '\n' inkem63tny2gg23tl5themdnl4yheyrrorpwk6dqoizxg435

Forty-eight characters. The character set — lowercase letters and digits 2-7 — was the unmistakable fingerprint of base32 encoding:

$ echo "INKEM63TNY2GG23TL5THEMDNL4YHEYRRORPWK6DQOIZXG435" | base32 -d The flag is a lieCTF{sn4cks_fr0m_0rb1t_expr3ss}

Snacks from orbit express. Delivered via DNS, one subdomain at a time.

What I Learned

This was a clean two-stage forensics problem. Stage 1 tested whether you could extract credentials from unencrypted HTTP traffic — a fundamental network analysis skill. Stage 2 tested whether you could recognize and decode DNS exfiltration, a technique that's been used in real-world breaches for over a decade.

The key takeaways:

  • Base64 is encoding, not encryption. If your authentication traffic isn't over TLS, base64-encoded credentials are readable by anyone on the wire. They might as well be plaintext.
  • DNS exfiltration hides in plain sight. DNS traffic is almost never blocked or inspected. Attackers encode stolen data into subdomain labels and query them against controlled nameservers. The data leaves the network disguised as routine name resolution.
  • Base32 is the encoding of choice for DNS tunneling. DNS labels are case-insensitive and limited to alphanumeric characters plus hyphens. Base32's character set (A-Z, 2-7) fits perfectly within those constraints, making it the standard encoding for DNS exfil payloads.
  • Sequence numbers make reassembly trivial. The s00-s11 prefixes eliminated any ambiguity about packet ordering. In real-world exfiltration, timing analysis or TCP sequence correlation might be needed instead.
