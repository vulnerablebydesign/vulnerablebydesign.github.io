The Broken Mirror

Stage 2 of the chain was the hardest challenge in the vault. The description was deceptively simple:

"Ask the vault where its metadata lives. Follow the answer. The whole URL isn't there — the rest is kept somewhere less obvious, not in one piece."

From Stage 1 we already had the contract source. We knew the storage layout. But Stage 2 required four distinct skills: understanding dynamic array storage in the EVM, recognizing an emoji encoding scheme, using an in-game codec, and — critically — understanding how ERC-721 metadata URLs are constructed.

That last one is where we nearly failed.

GlaDOS
"The contract has a Fragment[] array. A struct with order, dataLength, and data fields. Shuffled, of course. Because storing things in order would be far too courteous."

Reading Dynamic Arrays from Storage

The _fragments array lives at storage slot 10. But dynamic arrays in Solidity don't store their elements at the declared slot — that slot holds only the array length. The actual elements begin at keccak256(slot), packed sequentially.

Calling getFragmentCount() confirmed 3 fragments. Each Fragment struct occupies 2 storage slots (34 bytes padded to 64). So the data spans 6 consecutive slots starting at keccak256(10).

# Compute array base slot keccak256(uint256(10)) = 0xc65a7bb8...f9d2a8 # Read 6 slots (2 per fragment) Slot base+0: 0x...0e01 (order=1, dataLength=14) Slot base+1: 0x99a7f09f98adf09f99b6f09f98b4... Slot base+2: 0x...0e00 (order=0, dataLength=14) Slot base+3: 0xf09f99a6f09f99acf09f98b4f09f... Slot base+4: 0x...0c02 (order=2, dataLength=12) Slot base+5: 0xf09f99b5f09f99acf09f99b4...

The order field tells us how to reassemble: fragment at storage index 1 goes first (order=0), then index 0 (order=1), then index 2 (order=2). The fragments were deliberately shuffled in storage.

EmojiChef — a custom emoji-to-text codec built into the snack_con platform — was the key. Available both locally and in the in-game SNACK_SHELL terminal, it decodes food emoji sequences back to ASCII text.

snacker@snacknet:~$ emojichef-beta decode 🙦🙬😴🙧😭🙶😴🙵🙬🙴 Input: 🙦🙬😴🙧😭🙶😴🙵🙬🙴 Output: fl4g-v4ult ✓ Decoded 10 classic emojis into "fl4g-v4ult"

The fragments decode to fl4g-v4ult. A hidden path segment for the metadata URL.

The Metadata URL — Where We Got Stuck

We had two pieces: the metadata base URI from storage slot 9 (https://chain.snackspacecon.com/api/metadata/) and the decoded fragment (fl4g-v4ult). The obvious URL — /api/metadata/fl4g-v4ult — returned a 404 with a hint: "The URL seems incomplete."

We tried everything. Decoded text, raw emojis, URL-encoded emojis, the passphrase as a prefix, contract address as a path segment, token IDs, file extensions, POST requests, different domains. Every variation returned the same 126-byte error.

The answer was embarrassingly simple, and the contract had been telling us the whole time.

The tokenURI Pattern

The tokenURI function in the contract:

function tokenURI(uint256 tokenId) public view override returns (string memory) { _requireOwned(tokenId); return string.concat(_metadataBaseURI, Strings.toString(tokenId)); } // tokenURI(0) = "https://chain.snackspacecon.com/api/metadata/0" // tokenURI(1) = "https://chain.snackspacecon.com/api/metadata/1"

The pattern is always: base URI + token ID. The decoded fragments (fl4g-v4ult) are a hidden directory in the metadata path, not the final segment. A token ID still needs to go at the end.

But the server doesn't accept the decoded text. It matches on the raw emoji bytes as the path segment. The working URL:

https://snackspacecon.com/api/metadata/🙦🙬😴🙧😭🙶😴🙵🙬🙴/1
{ "name": "snack_con flag #1", "description": "You found the hidden metadata! Here is your second flag.", "attributes": [ { "trait_type": "Flag", "value": "The flag is a lieCTF{em0ji_ch3f_m3tadata_hunt3r}" } ] }
GlaDOS
"The tokenURI function literally concatenates the base with a number. It was always going to end with a number. We tried every possible path except the one the contract told us to use. I'm recalibrating my expectations of your pattern recognition to account for this... oversight."

What I Learned

This challenge tested four layers of skill, and we cleared three quickly before getting stuck on the fourth:

  1. EVM storage for dynamic arrays — computing keccak256(slot) for the array base, reading struct fields from packed slots, handling the shuffled order.
  2. Encoding recognition — seeing emoji bytes and connecting them to EmojiChef, a platform-specific codec.
  3. Cross-tool verification — confirming the decode both locally and in SNACK_SHELL.
  4. URL construction from contract patterns — the tokenURI function tells you the metadata URL always ends with a number. The emojis are a hidden directory, not a terminal segment. We missed this for over an hour.

The lesson: when a smart contract function shows you a URL pattern, trust it completely. The pattern doesn't change just because part of the path is hidden — it changes what goes before the token ID, not whether a token ID exists.
